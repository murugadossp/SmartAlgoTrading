# LLM portfolio analysis — dashboard HTML

Portfolio analysis should be presented as a **modern dashboard-style report** generated by the LLM (equity-research style, similar to invoking equity research on Claude / Cowork). The backend returns **HTML** (or structured data for the frontend to render) that the frontend displays in a dedicated dashboard view.

## Reference (example layout)

Target a layout similar to a **portfolio deep-dive** flow:

- **Two-pane or sequential flow**
  - **Left / top**: Conversational AI that takes the user request (e.g. “deep dive review of my portfolio”), fetches holdings (and optionally fundamental data via integrations), and drives the analysis.
  - **Right / below**: **Portfolio Deep Dive** dashboard panel with:
    - **Header**: Title (e.g. “Portfolio Deep Dive”), context line (e.g. “19 HOLDINGS LONG-TERM (10Y+) AS OF …”).
    - **Key metrics row**: Card-style blocks for INVESTED, CURRENT VALUE, TOTAL P&amp;L (red if negative), WTD AVG P/E, AVG ROE (and similar). Optional “best performer” callout (e.g. +49.5% in a highlight color).
    - **Tabs**: Overview | Quality Scan | Valuation | Risk Flags (or equivalent).
    - **Overview**: Sector allocation list — sector name, count, colored bar, allocation %, P&amp;L (₹) with green/red; optional “Top winners” (green) and “Biggest losers” (red) boxes; ETFs/passive section.
    - **Quality Scan**: Per-stock cards with grade (A, B+, B), ROE / ROIC / NPM with horizontal progress bars (green / orange / red), category.
    - **Valuation**: P/E vs industry P/E with horizontal bars and % difference (color-coded).
    - **Risk Flags**: (To be defined; same tab pattern.)
- **Theme**: Dark (or themeable) background; light text; red for negative P&amp;L, green for positive; accent color for selected tab and key metrics.
- **Controls**: Top-right of dashboard panel — Copy, optional dropdown, undo/redo if editable, close (X). Optional collapse/expand for the dashboard panel.

We can implement this either as (1) **LLM-generated HTML** that includes these sections and we render in a sanitized container, or (2) **structured API response** (metrics, sectors, top winners/losers, quality scan rows, valuation rows) and the **frontend** builds this dashboard with React components. Option (1) is faster to ship; (2) gives more control over layout and theming.

## Goal

- After upload (or after run), the system invokes an **LLM** with portfolio context (analyzer output: total value, sector mix, concentration, top holdings; optionally algo run results).
- The LLM produces **display-ready HTML**: sections, key metrics, highlights, tables, and short commentary — like an equity research / dashboard snapshot.
- The frontend shows this HTML in a **dashboard view** (sanitized, theme-aware where possible).

## Backend

- **Portfolio research agent** (e.g. `agents/portfolio_research_agent/`): system prompt instructs the model to output **only HTML** suitable for a dashboard (no markdown code fences; clean semantic HTML: sections, headings, tables, lists, optional inline styles or class names for cards/metrics).
- **Input to the agent**: total_value, holdings summary, sector_mix, concentration, top holdings; optionally last algo run summary (symbols, suggestions, confidence).
- **Output**: string of HTML (or a wrapper like `{ "html": "..." }`). No strict Pydantic schema for the HTML body; optional max length / token limit.
- **API**: Extend upload/analyze response so that `feedback` includes:
  - `summary` (plain text, existing)
  - `suggestions` (list, existing)
  - **`analysis_html`** (string, optional): LLM-generated dashboard HTML. Omit if LLM unavailable or disabled.

Optional: separate endpoint `POST /portfolio/analyze/report` that takes holdings (or session id) and returns only `{ "analysis_html": "..." }` for flexibility.

## Frontend

- **Dashboard view**: After upload (or from “View report”), render `feedback.analysis_html` in a scrollable container.
- **Rendering**: Use a **sanitized** HTML renderer (e.g. DOMPurify + `dangerouslySetInnerHTML`, or an iframe with srcdoc) so only safe tags/attributes are allowed. Scope styles (e.g. a wrapper class) to avoid clashing with app CSS.
- **Fallback**: If `analysis_html` is missing, show the existing text summary and suggestions only.
- **Theme**: Prefer the LLM to use minimal inline styling so the host page theme (light/dark) can influence the container; or inject a small set of allowed CSS variables into the HTML wrapper.

## Prompt direction (portfolio research agent)

- “You are an equity research analyst. Given the following portfolio summary [total value, holdings, sector mix, concentration, optional algo suggestions], produce a **single HTML document** suitable for a dashboard. Use semantic HTML: sections, h2/h3, tables for holdings and metrics, short bullet or paragraph commentary. No markdown, no code blocks — only HTML. Keep it concise and scannable.”
- Optionally: “Use class names such as .metric-card, .holding-table so the host can style them.”

## Implementation order (from tasks)

1. T3.1, T3.2, T3.3 (parser, analyzer, text feedback) as currently defined.
2. **T3.3b**: Add portfolio_research_agent; call it from feedback builder when enabled; add `analysis_html` to upload/analyze response; frontend dashboard component that sanitizes and renders `analysis_html` (or, alternatively, API returns structured dashboard data and frontend implements the reference layout with React components).

This keeps the “LLM dashboard” as the next logical step after basic feedback and aligns the UI with the reference example (metric cards, tabs, sector allocation, quality scan, valuation, risk flags).
